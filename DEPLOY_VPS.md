# TrashPanda VPS deployment

End-to-end runbook for deploying the FastAPI backend on RackNerd (or
any Debian/Ubuntu VPS) and pointing your local Next.js dev frontend
at it via an auto-restarting SSH tunnel.

## Topology

```
Browser → http://localhost:3000 (Next.js dev)
       → Next.js BFF (TRASHPANDA_BACKEND_URL=http://localhost:8001)
       → ssh -L 8001:127.0.0.1:8000 root@<vps_ip>     [auto-restarted]
       → uvicorn app.server:app on 127.0.0.1:8000     [systemd]
       → SMTP egress from RackNerd
       → /root/trashpanda/runtime/jobs/<id>/
```

## One-time install (on the VPS, as root)

```bash
ssh root@<vps_ip>
curl -fsSL https://raw.githubusercontent.com/AlexanderAtlasLuthor/TrashPanda/main/deploy/install_vps.sh -o install.sh
bash install.sh
```

What the installer does:

- Installs `git`, `python3`, `python3-venv`, `python3-pip`.
- Clones the repo into `/root/trashpanda`.
- Builds a virtualenv at `/root/trashpanda/.venv` and installs
  `requirements.txt` + `uvicorn`.
- Generates a fresh operator bearer token and writes
  `/etc/trashpanda/backend.env` (mode 640).
- Installs the systemd unit `trashpanda-backend.service` and enables
  it so the backend survives reboot.
- Smoke-checks `/healthz` and prints the operator token.

**Copy the printed `TRASHPANDA_OPERATOR_TOKEN` into your laptop's
`trashpanda-next/.env.local` so the BFF can authenticate.**

## Update the VPS to the latest commit

```bash
ssh root@<vps_ip> 'bash /root/trashpanda/deploy/update_vps.sh'
```

The update script pulls the latest `main`, refreshes deps, reinstalls
the systemd unit if it changed, restarts the service, and waits for
`/healthz`. Exit code is non-zero on failure so you can chain it in CI.

To pin a specific branch (e.g. for a hotfix preview):

```bash
ssh root@<vps_ip> 'REPO_BRANCH=claude/reduce-bounce-rate-ZJSLz bash /root/trashpanda/deploy/update_vps.sh'
```

## Environment file (operator-tunable knobs)

`/etc/trashpanda/backend.env`:

```env
# Bearer token enforced on every /api/operator/* request.
TRASHPANDA_OPERATOR_TOKEN=<auto-generated by installer>

# Optional: list a second token here to rotate without downtime.
# Both tokens are accepted simultaneously while operators upgrade.
# TRASHPANDA_OPERATOR_TOKENS=

# Wall-clock ceiling per job (seconds). 0 disables.
TRASHPANDA_MAX_JOB_SECONDS=2700

# Surfaced in the UI badge and /system/info.
TRASHPANDA_DEPLOYMENT=vps
TRASHPANDA_BACKEND_LABEL=racknerd
```

Any change to the env file requires a service restart:

```bash
sudo systemctl restart trashpanda-backend
```

## Auto-restarting SSH tunnel (on your laptop)

### Pre-flight: passwordless SSH

The tunnel supervisor cannot prompt for a password — it has to
reconnect on its own when the link drops. Set up key-based auth
*once* before running the tunnel:

**Windows (one-shot helper):**

```powershell
cd C:\path\to\TrashPanda
.\deploy\setup_ssh_key.ps1 -VpsHost root@192.3.105.145
```

The helper generates `~/.ssh/id_ed25519` if missing, asks for the
VPS password **once**, appends the public key to the VPS's
`/root/.ssh/authorized_keys`, and verifies that `ssh root@<vps>`
no longer prompts. Idempotent — safe to re-run after key rotation.

**macOS / Linux:**

```bash
ssh-keygen -t ed25519       # if you don't already have a key
ssh-copy-id root@192.3.105.145
ssh root@192.3.105.145 'echo "passwordless OK"'   # should NOT prompt
```

Until passwordless ssh works, `tunnel.ps1` / `tunnel.sh` will hang
on the password prompt and you'll see the supervisor loop without
ever bringing the tunnel up.

### Three tunnel options; pick the one that fits your OS.

### Option A — Windows (PowerShell supervisor)

```powershell
cd C:\path\to\TrashPanda
.\deploy\tunnel.ps1 -VpsHost root@192.3.105.145
```

Leave the window open. Logs to `%TEMP%\trashpanda-tunnel.log`.

### Option B — macOS / Linux (foreground supervisor)

```bash
cd ~/code/TrashPanda
VPS_HOST=root@192.3.105.145 bash deploy/tunnel.sh
```

Auto-detects `autossh` and uses it when present. Falls back to plain
`ssh` otherwise. Logs to `$TMPDIR/trashpanda-tunnel.log`.

### Option C — macOS / Linux (systemd-user unit, RECOMMENDED for headless)

For a tunnel that survives logout and reconnects on its own without
keeping a terminal open:

```bash
mkdir -p ~/.config/systemd/user
cp deploy/trashpanda-tunnel.service ~/.config/systemd/user/
# (Optional) override the VPS host with a drop-in:
systemctl --user edit trashpanda-tunnel
# Add:
#   [Service]
#   Environment=TRASHPANDA_VPS_HOST=root@your.vps.ip

systemctl --user daemon-reload
systemctl --user enable --now trashpanda-tunnel

# Check status / logs:
systemctl --user status trashpanda-tunnel
journalctl --user -u trashpanda-tunnel -f
```

The unit prefers `autossh` when installed. Install it with
`apt install autossh` (Debian/Ubuntu) or `brew install autossh`
(macOS) for the most aggressive dead-peer detection.

All three options use `ServerAliveInterval=10` /
`ServerAliveCountMax=3` so a dead peer is detected within ~30s, then
reconnects automatically.

## Frontend `.env.local`

Copy `trashpanda-next/.env.local.example` to `.env.local` and set:

```env
TRASHPANDA_BACKEND_URL=http://localhost:8001
TRASHPANDA_OPERATOR_TOKEN=<paste the token from install_vps.sh>
```

Restart `npm run dev` so Next.js picks up the new env vars.

## Verifying it works

After install + tunnel + frontend are all running:

```bash
# 1. Direct on the VPS:
curl -s http://127.0.0.1:8000/healthz | python3 -m json.tool

# 2. From your laptop (through the tunnel):
curl -s http://localhost:8001/healthz

# 3. Verify auth blocks unauthenticated operator calls:
curl -s -o /dev/null -w '%{http_code}\n' http://localhost:8001/api/operator/preflight
# expected: 401

# 4. Verify auth succeeds with a token:
curl -s -H "Authorization: Bearer $TOKEN" \
     http://localhost:8001/api/operator/jobs/missing
# expected: 200 / 404 — anything except 401/403 means auth passed.

# 5. Check the UI badge.
# Open http://localhost:3000 — top right should show:
#   "VPS · racknerd 🔒"   (when token is configured + tunnel up)
#   "Backend unreachable" (when tunnel drops)
```

## Operations

```bash
# Tail backend logs (on VPS)
sudo journalctl -u trashpanda-backend -f

# Restart the backend
sudo systemctl restart trashpanda-backend

# Disable / re-enable
sudo systemctl disable --now trashpanda-backend
sudo systemctl enable --now trashpanda-backend

# Show what runtime artifacts are on disk
ls -la /root/trashpanda/runtime/jobs/

# Run the Extra Strict Offline cleaner inside the VPS
cd /root/trashpanda
./.venv/bin/python scripts/extra_strict_clean.py --job-id <job_id>
```

## Security checklist

- [ ] `/etc/trashpanda/backend.env` is mode 640, owned by root.
- [ ] `TRASHPANDA_OPERATOR_TOKEN` is at least 32 random bytes
      (the installer uses `secrets.token_urlsafe(32)`).
- [ ] Backend binds to `127.0.0.1:8000` only — no public exposure.
- [ ] Port 8000 is **not** open in the VPS firewall to the internet.
- [ ] SSH access is key-based; password auth disabled on RackNerd.
- [ ] When rotating the token, list the new one in
      `TRASHPANDA_OPERATOR_TOKENS=`, restart the service, update the
      operator's `.env.local`, then drop the old token from
      `TRASHPANDA_OPERATOR_TOKEN`.

## Troubleshooting

| Symptom | Likely cause | Fix |
| --- | --- | --- |
| Topbar shows "Backend unreachable" | Tunnel dropped | Restart `deploy/tunnel.ps1`/`tunnel.sh` |
| `/api/operator/*` returns 401 | BFF has no token | Set `TRASHPANDA_OPERATOR_TOKEN` in `.env.local` and restart `npm run dev` |
| `/api/operator/*` returns 403 | Token mismatch | Re-read token from `/etc/trashpanda/backend.env`; update `.env.local` |
| Job stays "running" forever | Tunnel down + watchdog disabled | Confirm `TRASHPANDA_MAX_JOB_SECONDS > 0` in env file; restart service |
| `npm run dev` shows mock data | `TRASHPANDA_BACKEND_URL` unset | Copy `.env.local.example` → `.env.local`, set `http://localhost:8001` |
| `systemctl status` shows OOM | Large job exceeded `MemoryMax` | Increase `MemoryMax` in `deploy/trashpanda-backend.service` |
